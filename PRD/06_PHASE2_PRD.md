## 2. 기능 명세

### 2.1 분석 탭 5종 확장

옵션 상세 화면(또는 결정 상세 화면)의 분석 탭을 다음 5개로 구성:

1. 장단점 (Pros/Cons) — Phase 1 기존
2. 비교표 + 가중치 점수 (Comparison) — Phase 1 기존
3. SWOT — Phase 1 기존
4. **데카르트 4분면 (Cartesian Quadrants)** — 신규
5. **피터 드러커 5질문 (Drucker's 5 Questions)** — 신규

탭 순서는 위와 동일. 모바일에서는 가로 스크롤 또는 드롭다운으로 전환.

#### 2.1.1 데카르트 4분면

**개념:** 두 축의 조합으로 결정의 **모든 가능성 공간**을 4사분면에 매핑.
- 가로축: "이 일이 일어나면 / 일어나지 않으면"
- 세로축: "어떤 일이 일어날까 / 일어나지 않을까"

**4사분면:**

| 사분면 | 질문 | 의미 |
|---|---|---|
| 1사분면 (우상) | 이 일이 **일어나면** 어떤 일이 **일어날까?** | 실행 시 얻는 것 |
| 2사분면 (좌상) | 이 일이 **일어나지 않으면** 어떤 일이 **일어날까?** | 안 했을 때 생기는 것 |
| 3사분면 (좌하) | 이 일이 **일어나지 않으면** 어떤 일이 **일어나지 않을까?** | 안 했을 때 못 얻는 것 |
| 4사분면 (우하) | 이 일이 **일어나면** 어떤 일이 **일어나지 않을까?** | 실행 시 잃는 것 |

**UI 구성:**
- 상단: 2×2 그래프. 각 사분면에 **0–5개 키워드**(짧은 명사구) 표시. **사분면당 최대 5개**.
- 하단: 사분면 4개를 카드 형태로 펼쳐서 **각 키워드별 상세 설명** 1–2 문장.
- 키워드는 AI 생성, 사용자가 추가/삭제/수정 가능.
- 결정 단위 또는 옵션 단위로 분석 가능 (옵션이 2개 이상이면 옵션별 탭 전환).

**AI 생성 가이드:** "해당 사분면에 적합한 항목이 없으면 빈칸으로 두세요"는 가능하지만, **가급적 0개로 두지 말 것** — 빈약한 키워드라도 1–2개 제시하도록 프롬프트 작성. 4분면이 모두 채워질 때 분석 가치가 가장 큼.

**점수 영향(2.2 참조):** Q1 + Q3 항목 수 → 양(+), Q2 + Q4 → 음(−). 정규화는 §2.2 산정 룰의 비율 기반 산식 따름.

#### 2.1.2 피터 드러커 5질문

**개념:** 의사결정의 **본질**에 대한 5가지 자기 질문에 AI가 결정/맥락 기반으로 초안을 제시하고, 사용자가 자기 답변으로 다듬는다.

**5질문:**

| # | 질문 | 입력 형태 |
|---|---|---|
| 1 | **미션** — 나는 왜, 무엇을 위해 존재하는가? 나의 소명은 무엇인가? 지금 하려는 일은 그 소명에 부합하는가? | 자유 텍스트 + AI 초안 |
| 2 | **고객** — 나는 누구를 만족시키고 싶은가? | 자유 텍스트 + AI 초안 (자기/타인/다수 자동 분류) |
| 3 | **고객가치** — 그들이 (혹은 내가) 추구하고 가치 있게 여기는 것은 무엇인가? | 자유 텍스트 |
| 4 | **결과** — 어떤 결과를 바라며 그 결과는 나에게 어떤 의미가 있는가? | 자유 텍스트 |
| 5 | **계획** — 그래서 앞으로 무엇을 어떻게 할 것인가? | 자유 텍스트 (옵션 선택과 연계) |

**UI 구성:**
- 5질문을 카드 5장(혹은 단일 폼). 각 질문 아래에 AI 초안 + 사용자 답변 필드.
- 저장 시 결정 단위 1세트(옵션별 X — 옵션 무관하게 결정 자체에 대한 본질 질문).
- 답변은 §2.3 사용자 경향성 학습의 핵심 입력값(2번 답변 → 자기/타인 지향, 1·5번 답변 → 미션 부합도 점수).

### 2.2 추천 점수 (옵션 상단 표시)

#### 2.2.0 톤과 점수 모드 (Phase 2 재정의)

Phase 1의 톤(논리/감성/냉철)은 단순 어투 변경이었지만, Phase 2부터는 **점수 모드와 직결**된다. 사용자가 자기 성향에 갇히는 에코 챔버를 방지하기 위함.

| 톤 | 점수 모드 | 의미 | AI 설명 어투 |
|---|---|---|---|
| **논리** | `logical` | 성향 **무관** 베이스라인 — 기본 가중치 35/30/15/10/10 그대로, UserTendency 미반영 | 균형, 사실 중심 |
| **감성** | `emotional` | 성향 **반영** — UserTendency로 가중치 조정 | 공감, 개인화 |
| **냉철** | (점수 없음, 메타 뷰) | `logical`과 `emotional` 점수의 갭을 LLM이 설명 | 직설, 자기 점검 유도 |

**기본 추천 = 감성 점수.** 결정 상세 화면의 옵션 카드 점수/색상/박빙 뱃지/신뢰도 라벨은 모두 감성 점수 기준. 톤 드롭다운 변경 시:
- "논리"로 변경 → 동일 옵션의 logical 점수 + 객관 prose가 카드/탭에 다시 렌더링
- "냉철"로 변경 → 갭 설명 메타 뷰 표시 (점수 카드는 감성 점수 유지, 별도 섹션에 갭 분석)

**갭 알림 배너 (감성 모드 기본 화면에서):** 다음 중 하나라도 만족하면 옵션 카드 위에 부드러운 배너 표시:
- 논리 1위 옵션 ≠ 감성 1위 옵션
- 어떤 옵션의 `|logical_score − emotional_score| > 15`

배너 문구 예: "옵션 A는 객관 점수 82, 성향 반영 점수 64예요. 왜 차이 나는지 [냉철] 탭에서 확인하세요."

> **저장:** RecommendationScore 테이블이 `mode`(logical/emotional) 컬럼을 갖고 옵션당 모드별 1개 row. 냉철 갭 설명은 Decision.divergence_cache JSONB에 lazy 캐시 (§3.1, §3.2).

---

**위치:** 옵션 상세 화면 최상단(분석 탭 위쪽). 결정 상세에서는 모든 옵션 카드 상단.

**표기:**
- 0–100 정수 점수
- 색상:
  - **최고점 옵션 → 초록(연두)** — 장점 색상과 톤 매칭(Tailwind `green-500` 계열, 정확한 hex는 04_PROJECT_SPEC.md 디자인 가이드 따름)
  - **최저점 옵션 → 빨강** — 단점 색상과 톤 매칭(Tailwind `red-500` 계열)
  - **그 사이 → 회색** (`neutral-500`)
  - **박빙 룰**: 옵션이 2개 이상이고 **최고점과의 차이가 10점 이하**인 옵션이 ≥ 2개일 때 → 해당 옵션들 모두 **"박빙"** 뱃지 + 색상 약화(연한 톤). 단일 옵션이거나 명확한 1위가 있으면 미적용.
- 점수 옆에 **신뢰도 라벨**(2.3.5 임계치 기반):
  - 결정 < 5건 → "**초기 설정 기반**" (회색 배지) — 온보딩 자가 평가값 기준
  - 결정 5–14건 → "**학습 중**" (노랑 배지)
  - 결정 15건+ → "**경향성 반영**" (파랑 배지)

**산정 룰:**

각 옵션의 점수 = 다음 5개 시그널의 **가중 합**(각 시그널 0–100 정규화 후 가중치 곱하여 합산, 결과 0–100):

| 시그널 | 가중치 (기본) | 0–100 정규화 산식 |
|---|---|---|
| 장단점 균형 | 35% | `(장점 강도 합 / (장점 강도 합 + 단점 강도 합)) × 100`. 강도 = Claude가 항목별로 매긴 1–3 정수. 분모 0이면 50. |
| 비교표 가중 점수 | 30% | Phase 1 가중 합산 결과를 옵션 간 min–max 정규화. 옵션 1개거나 모두 동점이면 50. |
| SWOT 균형 | 15% | `((S 개수 + O 개수) / (S + W + O + T 개수)) × 100`. 분모 0이면 50. |
| 데카르트 4분면 부호 | 10% | `((Q1 + Q3) / (Q1 + Q2 + Q3 + Q4)) × 100`. 분모 0이면 50. |
| 드러커 미션 부합도 | 10% | Claude가 미션 답변과 옵션을 비교해 직접 0–100 산정. 미션 답변 비어있으면 50. |

> 데이터 없는 시그널은 항상 50(중립)으로 처리해 가중치 재배분 없이 일관성 유지.

**경향성 가중치 조정 (감성 모드 전용, 자가 평가 + 누적 데이터 기반, 단계 무관 항상 적용):**

> 논리 모드는 이 조정을 **건너뛴다** — 가중치 35/30/15/10/10 그대로 사용. 톤이 점수 모드를 결정한다(§2.2.0).

위 가중치 자체를 사용자 경향성 벡터로 조정:
- 분석적 사용자 → 비교표 가중치 ↑ (40%까지)
- 직관적 사용자 → SWOT/4분면 가중치 ↑
- 위험 회피 → 단점/W/T 항목에 음의 보정 ×1.2
- 위험 감수 → 장점/S/O 항목에 양의 보정 ×1.2
- 자기 지향 → 미션 부합도 가중치 ↑ (15%까지)

> 자가 평가만 있어도 경향성은 즉시 반영된다. 신뢰도 라벨이 단계별로 다른 이유는 **자가 평가의 잔여 영향력이 누적 데이터에 의해 자연 감쇠**하기 때문이지, 가중치 적용 여부 자체가 단계별로 다르지는 않다.

**주의 문구:**
- "초기 설정 기반" 단계:
  > 처음에 설정하신 성향 기준으로 매긴 점수예요. 결정이 쌓이면 실제 패턴으로 보정됩니다.
- "학습 중" 단계:
  > 자가 평가와 누적 결정을 혼합 중이에요. 결정 15건이 넘으면 본격 반영됩니다.

### 2.3 사용자 경향성 추적

**저장 위치:** `UserTendency` 테이블 (User 1:1, 5개 차원 + 메타 차원 1개).

**핵심 흐름:** 첫 로그인 시 **자가 평가 온보딩**으로 5축 초기값을 받고, 이후 결정/회고가 쌓이면 EWMA로 자동 보정. 콜드 스타트 문제(데이터 0건일 때 모두 중립 0)를 자가 평가로 해결.

#### 2.3.1 초기 자가 평가 (온보딩)

**시점:** 첫 로그인 직후, 첫 결정 작성 화면 진입 전.

**UI:** 슬라이더 5개짜리 단일 화면. 각 슬라이더 0–5 정수값.

| 차원 | 0 끝 라벨 | 5 끝 라벨 |
|---|---|---|
| 위험 감수도 | 매우 안전 추구 | 매우 도전적 |
| 시간 지향성 | 단기 보상 우선 | 장기 가치 우선 |
| 변화 수용도 | 보수적 | 진취적 |
| 분석 vs 직관 | 직관형 | 분석형 |
| 자기 vs 타인 | 자기 중심 | 타인 고려 |

**부가 안내:**
- "잘 모르겠으면 중간(2.5)에 두세요"
- "나중에 설정에서 다시 바꿀 수 있어요" (`/settings/tendency`에서 수정 가능)
- **건너뛰기 버튼** — 모든 슬라이더 2.5(중립)로 시작

**저장 변환:** 0–5 정수를 `(value − 2.5) / 2.5` 로 −1.0 ~ +1.0에 매핑하여 UserTendency 테이블에 기록. `decision_count = 0`으로 시작.

**메타 차원(만족 보정값) 제외:** 이 값은 회고 데이터에서만 누적되므로 자가 평가에 포함 안 함. 기본 3.0으로 시작.

#### 2.3.2 5개 차원 + 메타 1

| # | 차원 | 양 끝 | 입력 신호 |
|---|---|---|---|
| 1 | **위험 감수도** | 안전 (−1) ↔ 도전 (+1) | Claude가 결정 시 각 옵션을 "예측가능 / 불확실"로 라벨링 → 사용자가 최종 선택한 옵션의 라벨 누적 |
| 2 | **시간 지향성** | 단기 (−1) ↔ 장기 (+1) | Criteria 이름을 Claude가 "단기 보상 / 장기 가치"로 라벨링 → Criteria.weight 가중 평균 |
| 3 | **변화 수용도** | 보수적 (−1) ↔ 진취적 (+1) | Claude가 옵션을 "현상 유지 / 변화"로 라벨링 → 사용자 최종 선택의 라벨 누적 |
| 4 | **분석 vs 직관** | 직관 (−1) ↔ 분석 (+1) | Criteria 개수, 톤 재실행 횟수, 옵션당 평균 ProsConItem 개수, 결정 작성 평균 소요 시간 |
| 5 | **자기 vs 타인 지향** | 자기 (−1) ↔ 타인 (+1) | 드러커 Q2(고객) 답변에 본인/타인 비율 + 결정 context의 타인 언급 빈도 |
| meta | **만족 보정값** | 0.0–5.0 | Review.rating 평균. **Phase 2에서는 저장만**, 점수 산식에는 미반영. Phase 3 패턴 분석 대시보드 + 회고 텍스트 분석에서 활용. |

#### 2.3.3 업데이트 시점

- 새 결정 저장 시
- 새 회고 저장 시
- 결정의 final_choice가 변경될 때

#### 2.3.4 계산 방식

각 차원은 **지수 가중 이동 평균(EWMA)**, α = 0.2:
```
new_value = α × signal + (1 − α) × old_value
```
- `old_value`의 시작점 = 2.3.1의 자가 평가값 (`decision_count = 0`일 때)
- 최근 결정에 더 큰 가중치, 1년 이상 안 쓴 사용자도 과거 경향성 보존
- 자가 평가의 잔여 영향력: 결정 N건 후 약 `0.8^N` (예: 결정 10건 후 ≈ 11%, 20건 후 ≈ 1%)
  - → 사용자가 자기 성향을 잘못 답해도 실제 결정 패턴이 자연 보정함

#### 2.3.5 데이터 임계치

경향성은 **자가 평가가 입력된 순간부터 항상 점수에 반영**된다. 임계치는 **신뢰도 라벨**(사용자에게 보여주는 메시지)을 가르는 기준일 뿐, 가중치 적용 여부를 단계별로 끄고 켜지 않는다.

| 결정 수 | 경향성 벡터 구성 | 점수 라벨 (사용자 표시) |
|---|---|---|
| 0–4건 | EWMA 누적 거의 없음 → 사실상 자가 평가값 그대로 | "초기 설정 기반" |
| 5–14건 | EWMA가 자가 평가와 누적 데이터를 자동으로 섞음 (자가 평가 잔여 영향력 약 50%→33%) | "학습 중" |
| 15건+ | EWMA에서 자가 평가 잔여 영향력 약 5% 미만으로 자연 감쇠 | "경향성 반영" |

> 별도 단계 가중 블렌딩은 없다. EWMA(α=0.2)의 `0.8^N` 자연 감쇠가 단계별 표현을 만들어낸다.

#### 2.3.6 사용자 노출

설정 화면(`/settings/tendency`)에 "**나의 결정 성향**" 섹션:
- 5개 차원 레이더 차트 (자가 평가값 vs 현재값 둘 다 표시) — **5축 고정**, 만족 보정값 미포함
- 각 차원의 한 줄 설명 ("당신은 변화에 적극적인 편이에요")
- "이 분석이 점수에 어떻게 반영되는지" 토글
- **자가 평가 다시 하기** 버튼 — 슬라이더로 직접 재설정 가능 (decision_count는 보존)

만족 보정값은 **레이더 아래 별도 카드**로 표시:
- 카드 제목: "회고 만족도 평균"
- 본문: 평균 점수(소수점 1자리) + 누적 회고 수 + 한 줄 코멘트(예: "결정에 대체로 만족하고 계세요")
- 회고 0건일 때는 "아직 회고가 없어요. 결정 1주 후 알림이 가요" 안내로 대체
- **점수 산식과 무관**(2.3.2 메타 차원 참고) — 이 카드는 인사이트 표시용

> Phase 3의 "패턴 분석 대시보드"의 미니 버전. Phase 3에서 "후회한 결정 vs 만족한 결정" 같은 더 깊은 인사이트로 확장.

### 2.5 회고 알림

**채널 결정:** User 테이블의 `notification_channel` enum 필드 (`push` / `email` / `both` / `off`).

| 상황 | notification_channel 자동 설정 |
|---|---|
| 신규 가입 (PWA 미설치, 푸시 권한 없음) | `email` |
| 첫 결정 저장 직후 PWA 설치 + 푸시 권한 허용 | `push`로 자동 승격 (사용자 알림: "이제 푸시로 알려드릴게요") |
| 푸시 권한 거부 또는 PWA 미설치 유지 | `email` 유지 |

사용자는 설정에서 언제든 수동 변경 가능 (`push` / `email` / `both` / `off`). 수동 변경 후에는 자동 승격 로직 정지.

**스케줄:**
- decided_at + 7일 → "그 결정 어땠나요?" 1차 알림
- decided_at + 30일 → 2차 알림

**구현:** Supabase Edge Functions cron으로 일 1회 실행, 대상 Decision 추출 → 사용자 `notification_channel`에 따라 Push/이메일/둘 다 발송. `off`이면 skip.

**중복 발송 방지:** 알림 발송 직전에 해당 Decision의 `decided_at` 이후 Review 존재 여부 확인 — 이미 회고가 있으면 skip.

### 2.6 결정 링크 공유

- Decision에 `share_token` (UUID) 필드 추가, 기본 null
- "공유" 버튼 → 토큰 발급 + URL 복사 (`https://tiebreaker-weld.vercel.app/share/<token>`)

  - 표시: 결정 제목 + 옵션 + 비교표 + 추천 점수 + 최종 선택

- **OG 메타데이터**: 기본 ON(결정 제목만 노출, 옵션/맥락 제외). 사용자 옵션으로 OG 끄기 가능 — Phase 2 [NEEDS CLARIFICATION] 참고.

**RLS:** share_token이 not null인 Decision은 익명 SELECT 허용 (별도 정책).

### 2.8 PWA (모바일 앱 A안)

**범위:**
- `manifest.json` — 앱 이름, 아이콘(192/512), 테마 색
- Service Worker — 정적 자원 캐시 + offline fallback (분석은 온라인 필요, **히스토리 조회는 offline OK**)
- 홈 화면 추가 프롬프트 (iOS Safari + Android Chrome)
- Web Push 권한 요청 흐름 (회고 알림과 통합)
- **푸시 권한 요청 타이밍: 첫 결정 1건 저장 직후** — 가입 직후 요청하면 거부율 높음, 가치를 한 번 경험한 후 요청

**지원:**
- iOS Safari 16.4+ (Web Push 지원 시작)
- Android Chrome 전반

**제외 (네이티브 앱 — Phase 4):**
- App Store / Play Store 등록
- 네이티브 알림 사운드 / 위젯
- Face ID / Touch ID 인증

> **Phase 4 트리거:** PWA 설치 사용자가 50명 이상이고, "스토어에서 검색해서 깔고 싶다"는 요청이 3건 이상일 때 검토. 그 전까지는 PWA로 충분.

02_DATA_MODEL.md에 정의되어 있으나 0001/0002에 포함되지 않았다면 **0003에 같이 생성**. 이미 생성되어 있으면 스킵.

### 3.3 마이그레이션

Supabase 마이그레이션 0003 범위:
- **신규 테이블**: CartesianItem, DruckerAnswer, UserTendency, RecommendationScore, CriteriaTemplate
- **Review 테이블**: 0001/0002에 없으면 추가 (회고 알림이 의존)

- **시드**: CriteriaTemplate에 §2.7의 시스템 기본 4종 (`is_system = true`, `user_id = null`)
- **RLS 정책** (아래)

**RLS 정책:**
- CartesianItem, DruckerAnswer: Decision의 user_id 통해 검증
- UserTendency: `user_id == auth.uid()`
- RecommendationScore: Option → Decision → user_id 통해 검증
- CriteriaTemplate: `user_id == auth.uid()` OR `is_system = true` (읽기), 쓰기는 본인만

- 신규 사용자 가입 시 트리거: UserTendency row 자동 생성 (모든 차원 0, 자가 평가는 null)

## 5. AI 프롬프트 변경

04_PROJECT_SPEC.md의 프롬프트 가이드에 추가:

### 5.1 데카르트 4분면 생성 프롬프트

> 입력: 결정 제목, 맥락, 옵션
> 출력: 4사분면 × **0–5개 키워드** + 각 키워드 1–2문장 설명 (JSON)
> 가이드: "해당 사분면에 적합한 항목이 없으면 빈 배열 허용. 단 가급적 1–2개라도 채우도록 노력" (§2.1.1과 일치)

### 5.2 드러커 5질문 초안 생성 프롬프트

> 입력: 결정 제목, 맥락
> 출력: 5질문에 대한 초안 (각 100자 내외) — 사용자가 다듬을 수 있도록 단정 X, 질문형 어미 OK

### 5.3 옵션 라벨링 프롬프트 (경향성 입력)

> 입력: 옵션 + 결정 맥락
> 출력: `risk_label`(predictable/uncertain), `change_label`(status_quo/change), `time_label`(short/long)
> 결정 저장 시 백그라운드 호출, 결과는 `Option.ai_labels` JSONB 필드에 캐시 (§3.2 참조). 옵션 텍스트가 바뀌면 invalidate 후 재호출.

### 5.4 톤별 점수 산출 흐름

| 톤 | LLM 호출 | 가중치 | 캐시 위치 |
|---|---|---|---|
| 논리 | 5개 시그널 prose 생성(객관 어투) + score 산출 | 기본 35/30/15/10/10 | RecommendationScore.mode = `logical` |
| 감성 | 5개 시그널 prose 생성(공감 어투) + UserTendency 가중치 조정한 score 산출 | 경향성 조정값 | RecommendationScore.mode = `emotional` |
| 냉철 | (점수 X) 5.5 갭 설명 프롬프트 호출 | — | Decision.divergence_cache |

> 결정 저장 시 logical + emotional 양쪽 점수를 백그라운드에서 미리 계산. 사용자가 톤 변경 시 즉시 표시(prose는 기본 톤 캐시 외에는 lazy 생성).

### 5.5 냉철 갭 설명 프롬프트

> **입력:**
> - 옵션별 (logical_score, emotional_score, 5개 시그널 raw 값)
> - 사용자 UserTendency 5개 차원 + 자가 평가값
> - 갭이 큰 옵션 ID 리스트 (`|logical − emotional| > 15` 또는 1위 다름)
>
> **출력:** 갭이 큰 옵션마다 1–2문단:
> 1. 어떤 차원이 갭의 원인인지 ("당신은 위험 회피 성향(현재 −0.6)이라, 옵션 A의 '불확실' 라벨에 평균보다 1.2배 무겁게 페널티")
> 2. 자기 점검 질문 ("이 가중이 이번 결정에서도 합리적인가요?")
>
> **톤:** 직설, 비판적이지만 과격하지 않게. 사용자를 비난하지 않고 패턴을 짚어주는 어투.
>
> **금지:** "이 옵션을 선택하세요/마세요" 같은 직접 권유 문구. 냉철 톤은 판단 도우미가 아니라 **자기 인식 도우미**.
>
> **캐시:** Decision.divergence_cache JSONB. 점수 변경 시 invalidate.
